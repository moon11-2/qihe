# 后端开发交接文档

日期：2026-07-04  
项目名称：契合  
当前阶段：前端已跑通，本地 Dify 代理已验证，正式后端待建设

## 1. 文档用途

本文档给后端开发同学使用，说明为了让「契合」从本地前端原型进入可内测、可上线状态，后端需要实现哪些服务。

## 2. 后端目标

后端第一阶段目标不是做完整平台，而是提供一个安全、稳定、可被 H5/iOS 调用的 AI 代理层。

推荐架构：

```text
H5 / iOS WKWebView
  -> 自有 HTTPS 后端
  -> Dify App API
  -> 大模型 / 知识库 / 工作流
```

## 3. 为什么必须有后端

不能让前端直接保存或调用 Dify API Key，原因：

- App 包可被解包。
- 浏览器 JS 可被查看。
- Dify Key 泄露后会造成费用和数据风险。
- App Store 上架也不应把第三方密钥放进客户端。

因此正式环境必须由后端保存 Key，并由后端转发请求。

## 4. 第一阶段必须实现的接口

| 优先级 | 接口 | 说明 |
| --- | --- | --- |
| P0 | `GET /health` | 健康检查 |
| P0 | `POST /api/dify/chat-messages` | Dify Chat Messages 兼容代理 |
| P0 | `POST /api/dify/files/upload` | Dify 文件上传兼容代理 |

第一阶段可以先兼容 Dify 官方接口形态，减少前端改动。

## 5. 第二阶段建议实现的业务接口

| 优先级 | 接口 | 说明 |
| --- | --- | --- |
| P1 | `POST /api/v1/contracts/analyze` | 业务封装后的合同处理接口 |
| P1 | `POST /api/v1/files` | 业务文件上传接口 |
| P2 | `GET /api/v1/records` | 云端历史 |
| P2 | `POST /api/v1/records` | 保存历史 |
| P2 | `GET /api/v1/settings` | 用户设置 |
| P2 | `POST /api/v1/settings` | 更新设置 |

业务接口不是前端交接阶段必需，但正式产品化建议补齐。

## 6. 环境变量

正式后端至少需要：

```env
DIFY_API_BASE_URL=https://api.dify.ai/v1
DIFY_API_KEY=replace-with-server-side-key
PORT=3000
REQUEST_TIMEOUT_MS=180000
UPLOAD_MAX_SIZE_MB=20
ALLOWED_ORIGINS=https://your-domain.example
```

要求：

- `DIFY_API_KEY` 只能存在服务端。
- 不得返回给前端。
- 不得写入日志。
- 不得提交到 Git。
- 生产环境必须使用 HTTPS。

## 7. Dify 请求代理要求

前端当前会发送：

```http
POST /api/dify/chat-messages
Content-Type: application/json
```

后端需要：

1. 校验请求来源。
2. 校验请求体大小。
3. 加上 Dify Authorization：

   ```http
   Authorization: Bearer ${DIFY_API_KEY}
   ```

4. 转发到：

   ```text
   https://api.dify.ai/v1/chat-messages
   ```

5. 返回 Dify 兼容响应。
6. 保留 `conversation_id`。
7. 不泄露 Key。

## 8. 流式响应建议

Dify blocking 请求可能耗时较长。建议后端内部使用 streaming 请求 Dify，再聚合成前端当前可解析的 blocking 结果。

本项目本地代理已经做了类似处理：

```text
scripts/dify-proxy.mjs
```

后端可参考其思路，但生产实现应补齐：

- 超时控制
- 错误码
- 日志脱敏
- 鉴权
- 限流
- 文件大小限制
- 监控

## 9. 文件上传代理要求

前端会上传：

- PDF
- DOCX
- TXT
- MD

后端需要：

1. 限制文件大小。
2. 限制文件类型。
3. 做基本安全检查。
4. 转发到 Dify 文件上传接口。
5. 返回 Dify `upload_file_id`。
6. 不长期保存用户合同文件，除非产品和隐私政策明确说明。

## 10. 日志和隐私

禁止记录：

- Dify API Key
- 模型供应商 Key
- 用户完整合同正文
- 身份证号
- 手机号
- 银行卡号
- 合同中的完整个人敏感信息

建议记录：

- 请求 ID
- 用户匿名 ID
- 接口路径
- 响应耗时
- 状态码
- 错误类型
- Dify conversation_id
- 文本长度
- 文件大小

## 11. 错误处理建议

建议后端统一错误形态：

```json
{
  "error": "dify_timeout",
  "message": "在线服务响应超时，请稍后重试。",
  "requestId": "req_xxx"
}
```

常见错误：

| 错误 | HTTP | 说明 |
| --- | --- | --- |
| `bad_request` | 400 | 请求字段缺失或格式错误 |
| `file_too_large` | 413 | 文件过大 |
| `unsupported_file_type` | 415 | 文件类型不支持 |
| `dify_timeout` | 504 | Dify 超时 |
| `dify_error` | 502 | Dify 返回错误 |
| `rate_limited` | 429 | 请求过于频繁 |
| `server_error` | 500 | 后端内部错误 |

## 12. 生产安全要求

上线前至少需要：

- HTTPS
- CORS 白名单
- 基础鉴权或匿名设备 ID 风控
- IP 限流
- 文件大小限制
- Dify Key 服务端保存
- 日志脱敏
- 请求超时
- 错误追踪
- 健康检查

## 13. 后端验收标准

后端第一阶段完成后，应满足：

1. H5 能通过正式 HTTPS 后端调用合同审查。
2. H5 能通过正式 HTTPS 后端调用合同生成。
3. 首页咨询能返回 Dify `consult`。
4. 文件上传可用。
5. 前端不暴露任何 Dify Key。
6. 失败时前端能显示可理解错误。
7. 请求日志不包含完整合同正文和敏感信息。

## 14. 不属于第一阶段的事项

以下事项可以后置：

- 完整用户系统
- 会员付费
- 云历史
- 管理后台
- 多租户
- 复杂统计报表
- 合同模板市场

## 15. 交接结论

当前后端工作可以从 `P0` 三个接口开始。只要完成安全代理、文件上传和健康检查，前端即可从本地代理切换到正式后端联调。
