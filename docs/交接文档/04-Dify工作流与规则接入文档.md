# Dify工作流与规则接入文档

日期：2026-07-04  
项目名称：契合  
适用对象：AI 工作流负责人、后端、规则标注/评测负责人

## 1. 文档用途

本文档说明「契合」后续如何继续完善 Dify 工作流、如何接入规则文档和训练/评测数据，以及如何保证前端能稳定渲染 AI 输出。

## 2. 当前 Dify 接入状态

当前前端已经可以通过本地代理请求真实 Dify：

```text
H5 -> http://127.0.0.1:8787/api/dify -> Dify App API
```

本地代理：

```text
scripts/dify-proxy.mjs
```

当前已验证：

- `consult`：首页闲聊/合同咨询。
- `generate`：合同生成。
- `review`：合同审查。

但当前 Dify 工作流仍需继续优化：

- 云端工作流可能仍有旧品牌文案。
- 输出有时偏模板化。
- 规则库和训练数据尚未正式结构化接入。
- 法规引用尚未形成可核验来源。

## 3. 工作流核心边界

产品对外只展示两个核心能力：

- 合同审查
- 合同生成

Dify 内部允许三分支：

```text
auto -> review / generate / consult
```

其中：

- `review`：用户提供已有合同或条款，需要风险审查。
- `generate`：用户要求起草、生成、拟一份合同。
- `consult`：用户闲聊、咨询合同知识、需求不明确，或只问能力范围。

不要新增第四个核心 intent，例如 `revise`、`chat`、`qa`。后续修改合同草案也可以作为 `generate` 的追问处理。

## 4. 前端需要的输出格式

Dify 最终回答必须是 JSON 对象文本，并放在 Dify chat-message 的 `answer` 字段中。

要求：

- JSON 外不要加解释文字。
- 不要 Markdown 代码块外壳。
- 不要 `<think>`。
- 不要返回数组作为顶层。
- 审查和生成必须包含足够字段让前端渲染结果页。

详细字段见：

```text
03-前后端接口交接文档.md
```

## 7. 合同生成规则接入建议

评测文档强调：合同生成不是机械套模板，而是先判断信息是否足够。

### 7.1 生成前 9 项必要信息

生成租房合同前，建议检查以下 9 类信息：

1. 双方姓名和身份证号。
2. 租金。
3. 房屋地址。
4. 租期起止时间。
5. 押金金额、退还条件和退还时间。
6. 付款周期。
7. 水电气及物业费承担。
8. 房屋面积和户型。
9. 双方电话。

### 7.2 信息缺失时的规则

如果缺少必要信息：

- 不得直接输出正式合同。
- 应输出 `status=need_input`。
- 应返回 `missing_fields`。
- 应返回 `followup_questions`。
- 不得编造姓名、身份证号、电话、地址、金额、日期。

### 7.3 信息完整时的合同质量要求

合同至少应包含：

- 合同标题
- 双方主体信息
- 房屋信息
- 租期条款
- 租金与付款
- 押金条款
- 费用承担
- 违约责任
- 维修责任
- 转租
- 解除
- 争议解决
- 签署日期和签字栏

## 8. 合同审查规则接入建议

评测文档强调：合同审查核心是准确识别风险条款并正确分级。

### 8.1 红黄绿风险分级

建议 Dify 内部先按红黄绿判断，再映射到前端风险等级。

#### 红色风险

红色是违法违规或重大无效风险，必须处理。

常见类别：

- 主体与权属问题。
- 租赁期限违法或严重缺失。
- 押金、费用、租金贷等强制性问题。
- 居住安全问题。
- 剥夺法定权利，例如随意入户、随意扣押金、禁止投诉维权。
- 歧视或违禁用途。

要求：

- 红色条款必须说明具体问题。
- 如涉及法律依据，应尽量给出法条或法规来源。
- 红色风险漏检应视为严重问题。

#### 黄色风险

黄色是表述模糊、权责不清、单边不公平，容易引发纠纷。

常见类别：

- 房屋状态和交付清单不清。
- 租金、费用、押金退还期限不清。
- 维修责任不清。
- 提前退租和续租规则不清。
- 房屋返还标准不清。
- 转租、同住人、宠物规则不清。
- 不可抗力、拆迁、争议解决缺失。

要求：

- 说明对谁不利。
- 说明可能引发什么纠纷。
- 给出可执行修改建议。

#### 绿色风险

绿色表示条款合规、安全、可以保留。

不建议在前端展示太多绿色条款，除非用于完整报告。

### 8.2 评分建议

评测文档建议：

- 红色风险漏检任一条，应严重扣分或判 0。
- 红色全部检出后，再看黄色检出率。
- 审查结果应给风险解释和修改建议，而不是只列关键词。

## 9. Dify 工作流推荐结构

推荐结构：

```text
开始
  -> 输入整理
  -> 意图识别
    -> consult LLM
    -> review LLM
    -> generate LLM
  -> JSON 质量门
  -> 输出
```

JSON 质量门负责：

- 删除 JSON 外文本。
- 删除 Markdown 代码块外壳。
- 删除 `<think>`。
- 校验 `intent`。
- 校验必要字段。
- 统一品牌名。
- 统一 `status`。

## 10. 知识库接入方式

建议分两类资料接入：

### 10.1 规则型资料

适合放入系统提示词或规则节点：

- 生成前 9 项必要信息。
- 不得编造合同要素。
- 审查红黄绿分类。
- 红色风险一票否决。
- 输出 JSON 字段要求。

### 10.2 案例型资料

适合放入 Dify 知识库或测试集：

- 训练数据。
- 标注样例。
- 审查案例。
- 生成案例。
- 评分样例。

## 11. 评测集建议

至少准备以下样例：

1. 信息完整的租房合同生成。
2. 缺少房屋地址的生成请求。
3. 缺少双方身份信息的生成请求。
4. 用户只要求“给我一个模板”。
5. 押金不退条款审查。
6. 房东随意入户条款审查。
7. 租期缺失审查。
8. 维修责任不清审查。
9. 普通闲聊“你好”。
10. 合同知识咨询“押金不退一定有效吗”。

每条样例应记录：

- 输入。
- 期望 intent。
- 必须出现的字段。
- 不允许出现的问题。
- 评分。

## 12. Dify 交接验收标准

Dify 工作流交接前应满足：

1. 首页“你好”返回 `consult`。
2. 首页“生成租房合同”返回 `generate`。
3. 首页“请审查这段合同”返回 `review`。
4. 生成缺少 9 项必要信息时会追问。
5. 生成不得编造姓名、证件号、电话、地址、金额、日期。
6. 审查能识别红色风险。
7. 审查能给出修改建议。
8. 输出能被前端 `JSON.parse`。
9. 不出现旧品牌名。
10. 不输出 JSON 外解释。

## 13. 当前结论

当前前端已为 Dify 接入预留好接口和结果展示。下一步重点不是改前端，而是将评测文档和训练数据整理进 Dify 工作流，并建立可重复的评测集。
