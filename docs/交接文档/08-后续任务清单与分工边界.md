# 后续任务清单与分工边界

日期：2026-07-04  
项目名称：契合

## 1. 文档用途

本文档用于项目交接后的任务拆分，明确哪些事情已经完成，哪些事情需要后续同学继续做，以及当前交付人的职责边界。

## 2. 当前交付人已完成

当前交付人主要完成：

- 前端 H5 原型。
- 移动端核心 UI。
- 首页自由输入。
- 合同审查流程。
- 合同生成流程。
- 历史和设置。
- Dify 本地代理联调验证。
- iOS wrapper 基础工程。
- QA 截图和测试记录。
- 交接文档。

## 3. 当前交付人不负责

明确不负责：

- 正式服务器搭建。
- 正式后端 API 开发。
- 生产数据库。
- 正式 Dify 工作流发布。
- App Store Connect 提交。
- 隐私政策网页上线。
- 用户协议网页上线。
- 发布会、视频、PPT、宣传材料。
- 付费、账号、云历史。

## 4. P0 后续任务

P0 是让产品能进入真实内测必须完成的任务。

### 4.1 后端 P0

负责人：后端同学

任务：

1. 搭建 HTTPS 后端。
2. 实现 `/health`。
3. 实现 `/api/dify/chat-messages`。
4. 实现 `/api/dify/files/upload`。
5. 服务端保存 Dify Key。
6. 日志脱敏。
7. 超时和错误响应。
8. CORS 白名单。
9. 文件大小限制。

验收：

- H5 可通过正式后端完成合同生成和审查。
- 前端包内没有 Dify Key。
- 后端日志没有完整合同正文。

### 4.2 Dify P0

负责人：AI 工作流同学

任务：

1. 清理旧品牌文案。
2. 固定三分支：`consult/review/generate`。
3. 输出合法 JSON。
4. 接入租房合同生成必要信息规则。
5. 接入租房合同审查红黄绿风险规则。
6. 建立 10 条以上测试样例。
7. 发布稳定版本。

验收：

- `你好` 返回 `consult`。
- 租房合同生成返回 `generate`。
- 租房合同审查返回 `review`。
- 信息缺失时追问，不编造。
- 审查能识别红色风险。

### 4.3 iOS P0

负责人：iOS/上线同学

任务：

1. 同步最新 H5。
2. 配置 Bundle ID。
3. 配置签名。
4. 真机运行。
5. 测试键盘、滚动、安全区。
6. 测试文件上传。
7. 测试复制和导出。

验收：

- iPhone 真机可完整完成合同生成和审查。
- 没有白屏。
- 没有输入框被键盘完全遮挡。

## 5. P1 后续任务

P1 是进入外部内测或上架前建议完成的任务。

### 5.1 产品合规

负责人：产品/法务/上线

任务：

1. 隐私政策。
2. 用户协议。
3. AI 免责声明。
4. 法律服务边界说明。
5. 支持邮箱或支持网页。

### 5.2 App Store 材料

负责人：上线/设计/市场

任务：

1. App 图标。
2. 商店截图。
3. App 简介。
4. 关键词。
5. 审核说明。
6. TestFlight 内测。

### 5.3 质量和监控

负责人：后端/测试

任务：

1. 错误监控。
2. 请求日志。
3. Dify 耗时统计。
4. 生成失败率。
5. 用户反馈入口。

## 6. P2 后续任务

P2 是产品后续增强。

任务：

- 账号登录。
- 云历史。
- 多设备同步。
- 会员/付费。
- 模板库。
- 法规知识库。
- 合同分类库。
- 管理后台。
- 标注评测平台。

## 7. 交接后推荐推进顺序

建议顺序：

1. 后端完成 P0 安全代理。
2. Dify 工作流按规则文档修正。
3. 前端切正式后端地址联调。
4. iOS 同步最新 H5 真机测试。
5. 补隐私政策和用户协议。
6. TestFlight。
7. App Store 上架材料。
8. 发布会/PPT/视频。

## 8. 风险提醒

当前最大风险不在前端，而在：

- Dify 输出质量是否稳定。
- 规则和训练数据是否真正接入。
- 后端是否能安全保存 Key。
- iOS 真机文件上传是否稳定。
- App Store 对 AI/法律类产品的审核说明是否充分。

## 9. 分工边界结论

当前前端阶段可以交接。后续工作应由后端、AI 工作流、iOS/上线、产品/发布材料负责人分别推进，不应再把服务器、上线、发布会材料归入前端交付范围。
