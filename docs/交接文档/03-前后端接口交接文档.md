# 前后端接口交接文档

日期：2026-07-04  
项目名称：契合  
适用对象：前端、后端、Dify 工作流联调人员

## 1. 文档用途

本文档定义当前前端和后端之间需要对齐的请求、响应、字段、枚举和错误处理。

第一阶段建议后端直接兼容 Dify 接口路径，减少前端改动。

## 2. 当前前端调用入口

前端统一调用：

```text
src/services/dify.ts
```

核心函数：

```ts
runContractAnalysis(payload)
```

当前前端会根据环境变量决定：

- `VITE_DIFY_ENABLED=false`：走 Mock。
- `VITE_DIFY_ENABLED=true`：请求 `VITE_DIFY_PROXY_PATH`。

正式后端接入时，建议把：

```env
VITE_DIFY_PROXY_PATH=https://api.your-domain.example/api/dify
```

或通过运行时配置注入，不要写死本机地址。

## 3. 业务模式

前端请求后端的 `mode` 只有三种：

```json
["auto", "review", "generate"]
```

含义：

| mode | 含义 |
| --- | --- |
| `auto` | 首页自由输入，由 Dify/后端判断审查、生成或咨询 |
| `review` | 明确进行合同审查 |
| `generate` | 明确进行合同生成 |

后端/Dify 返回的 `intent` 可以有三种：

```json
["review", "generate", "consult"]
```

其中：

- `review`：进入审查结果页。
- `generate`：进入生成结果页。
- `consult`：只用于首页聊天流，不是第三个核心功能。

## 4. 前端请求结构

前端内部 payload：

```ts
interface ContractAnalysisPayload {
  mode: "auto" | "review" | "generate";
  query: string;
  contractText: string;
  contractType: string;
  role: string;
  focusAreas: string[];
  requirements: string;
  jurisdiction?: string;
  outputStyle?: string;
  conversationId?: string;
  file?: {
    name: string;
    path: string;
    size?: number;
    type?: string;
  } | null;
}
```

前端发给 Dify 兼容接口的 JSON：

```json
{
  "inputs": {
    "mode": "auto",
    "contract_text": "用户原始输入或合同正文",
    "contract_type": "房屋租赁合同",
    "role": "甲方",
    "focus_areas": "付款风险，违约责任，争议解决",
    "requirements": "输出偏好：普通用户可读，结构清晰",
    "jurisdiction": "中国大陆",
    "output_style": "普通用户可读"
  },
  "query": "用户原始输入",
  "response_mode": "blocking",
  "conversation_id": "",
  "user": "qihe-h5-xxxxxx"
}
```

注意：

- `output_style` 当前应固定为 `普通用户可读`，避免 Dify 工作流枚举不匹配。
- 用户选择的“简洁/标准/详细”放入 `requirements` 中。
- `focusAreas` 前端数组会转成中文逗号字符串。

## 5. 字段说明

| 字段 | 来源 | 说明 |
| --- | --- | --- |
| `mode` | 前端 | `auto/review/generate` |
| `query` | 前端 | 用户原始输入 |
| `contract_text` | 前端 | 合同正文或生成需求 |
| `contract_type` | 前端 | 合同类型 |
| `role` | 前端 | 用户身份或立场 |
| `focus_areas` | 前端 | 审查重点或生成条款 |
| `requirements` | 前端 | 补充说明和输出偏好 |
| `jurisdiction` | 前端 | 默认中国大陆 |
| `output_style` | 前端 | 固定普通用户可读 |
| `conversation_id` | Dify/前端 | 多轮会话 ID |
| `user` | 前端生成 | 匿名本地用户 ID |

## 6. 合同类型枚举

建议后端兼容以下值：

```json
[
  "不确定",
  "买卖/采购合同",
  "房屋租赁合同",
  "服务/委托合同",
  "技术/软件合同",
  "劳动/用工合同",
  "工程/装修合同",
  "借款/担保合同",
  "合伙/合作合同",
  "自定义合同"
]
```

兼容旧值：

- `租赁/房屋合同` 等价于 `房屋租赁合同`
- `工程/装修合同` 可映射为建设工程、装修施工、承揽类

## 7. 用户身份枚举

当前前端：

```json
["甲方", "乙方", "中立", "未知"]
```

Dify/后端可以在内部扩展理解：

- `甲方`：我方、委托方、出租方、付款方等，需结合合同文本判断。
- `乙方`：对方、服务方、承租方、收款方等，需结合合同文本判断。
- `中立`：同时提示双方风险。
- `未知`：不要擅自假设。

## 8. 响应结构

后端建议保持 Dify chat-message 形态：

```json
{
  "event": "message",
  "task_id": "task-id",
  "id": "message-id",
  "message_id": "message-id",
  "conversation_id": "conversation-id",
  "mode": "advanced-chat",
  "answer": "{\"intent\":\"generate\",\"status\":\"complete\"}"
}
```

前端会解析 `answer` 字符串中的 JSON。

要求：

- `answer` 必须是字符串。
- 字符串内容必须能 `JSON.parse`。
- 不要在 JSON 外加解释文字。
- 不要返回 Markdown 代码块外壳。
- 不要返回 `<think>`。

## 9. `consult` 响应

用于首页聊天。

必要字段：

```json
{
  "intent": "consult",
  "status": "complete",
  "result_type": "contract_consultation",
  "title": "合同咨询",
  "summary": "简短摘要",
  "answer_markdown": "可展示给用户的回答",
  "followup_questions": ["可选追问"],
  "disclaimer": "AI 辅助说明不构成律师法律意见。"
}
```

## 10. `review` 响应

用于审查结果页。

必要字段：

```json
{
  "intent": "review",
  "status": "complete",
  "result_type": "review_report",
  "title": "合同审查报告",
  "contract_type": "房屋租赁合同",
  "role": "甲方",
  "score": 76,
  "grade": "C",
  "grade_label": "中等风险",
  "risk_level": "medium",
  "summary": "审查摘要",
  "clause_reviews": [],
  "facts": {},
  "missing_fields": [],
  "markdown_report": "# 审查报告",
  "disclaimer": "AI 辅助审查，不构成律师法律意见。"
}
```

前端结果页重点使用：

- `summary`
- `score`
- `grade`
- `risk_level`
- `clause_reviews`
- `facts`
- `markdown_report`

  "signing_checklist": [],
  "disclaimer": "AI 辅助起草，不构成律师法律意见。"

前端结果页重点使用：

- `summary`
- `contract_title`
- `contract_markdown`
- `missing_fields`
- `signing_checklist`
- `risk_notes`

## 12. 信息不足

如果信息不足，建议返回：

```json
{
  "intent": "generate",
  "status": "need_input",
  "summary": "请补充合同类型、双方身份、金额和期限。",
  "missing_fields": ["合同类型", "双方身份", "金额", "期限"],
  "followup_questions": ["你是甲方还是乙方？", "合同金额是多少？"]
}
```

前端会展示摘要和待补信息。

## 13. 错误响应

建议后端错误：

```json
{
  "error": "dify_timeout",
  "message": "在线服务响应超时，请稍后重试。",
  "requestId": "req_xxx"
}
```

错误码建议见 `02-后端开发交接文档.md`。

## 14. 联调步骤

1. 后端启动 `/health`。
2. 前端设置 `VITE_DIFY_ENABLED=true`。
3. 前端设置 `VITE_DIFY_PROXY_PATH` 指向后端。
4. 首页发送“你好”，应返回 `consult`。
5. 首页发送“生成一份租房合同...”，应返回 `generate`。
6. 首页发送“请审查这段合同...”，应返回 `review`。
7. 功能页合同生成应进入生成结果页。
8. 功能页合同审查应进入审查结果页。
9. 文件上传应返回 Dify file id。

## 15. 接口交接结论

只要后端保持当前 Dify 兼容接口和 `answer` JSON 契约，前端无需大改即可接入正式服务。
