## 4. Mock 模式

如果不接 Dify，可以使用 Mock 模式检查界面。

环境变量：

```env
VITE_DIFY_ENABLED=false
```

Mock 逻辑在：

```text
src/services/dify.ts
```

Mock 模式适合：

- 前端 UI 验收。
- iOS wrapper 视觉验证。
- 无 Key 环境演示。

Mock 模式不适合：

- 判断 AI 质量。
- 判断 Dify 工作流质量。
- 判断后端稳定性。

## 5. Dify 本地代理模式

本地 Dify 代理脚本：

```text
scripts/dify-proxy.mjs
```

启动：

```bash
npm run dify:proxy
```

默认监听：

```text
http://127.0.0.1:8787/api/dify
```

H5 dev server 需要注入：

```env
VITE_DIFY_ENABLED=true
VITE_DIFY_PROXY_PATH=http://127.0.0.1:8787/api/dify
```

注意：

- 本地代理会读取 `.env.local`。
- 不要提交 `.env.local`。
- 不要输出 Dify Key。
- 本地代理只用于开发联调，不是生产服务。

## 9. 正式部署建议

正式部署应改为：

```text
H5/iOS
  -> https://api.your-domain.example
  -> Dify
```

正式环境变量示例：

```env
DIFY_API_BASE_URL=https://api.dify.ai/v1
DIFY_API_KEY=server-side-key
REQUEST_TIMEOUT_MS=180000
UPLOAD_MAX_SIZE_MB=20
ALLOWED_ORIGINS=https://your-app-domain.example
```

前端只需要知道后端地址：

```env
VITE_DIFY_ENABLED=true
VITE_DIFY_PROXY_PATH=https://api.your-domain.example/api/dify
```

更推荐后续改成运行时配置，而不是重新打包前端。

## 10. 部署检查清单

正式部署前检查：

- HTTPS 可用。
- `/health` 可用。
- `/api/dify/chat-messages` 可用。
- `/api/dify/files/upload` 可用。
- Dify Key 不暴露。
- CORS 白名单正确。
- 请求超时足够长。
- 日志脱敏。
- 错误响应前端可展示。
- 文件大小限制生效。

## 11. 常见问题

### 11.1 页面像没有接 AI

检查：

- `VITE_DIFY_ENABLED` 是否为 `true`。
- 页面加载到的 `src/services/dify.ts` 是否注入了正确环境变量。
- `VITE_DIFY_PROXY_PATH` 是否指向可用代理。
- 代理日志是否有 `POST /api/dify/chat-messages -> 200`。
- Dify 返回的 `answer` 是否为空。

### 11.2 Dify 返回空 answer

曾发现 `output_style` 传不符合工作流枚举时可能导致空返回。当前前端已固定传：

```text
普通用户可读
```

后续工作流如调整枚举，应同步更新接口文档。

### 11.3 宽屏抽屉偏移

已修复。原因是固定抽屉的居中 transform 被动画覆盖。当前宽屏下抽屉通过固定 left 计算对齐手机壳。
