# 合同帮 APP 项目计划书

版本：v0.2  
日期：2026-07-01  
状态：内部测试版规划与视觉探索阶段

## 1. 项目定位

“合同帮”是一款面向手机端的 AI 合同审查与合同生成 APP。第一版目标不是商业化上线，而是做一个 20 人以内可试用的内部测试版，用来验证交互、合同处理能力和 Dify 工作流质量。

核心能力：

- 合同审查：输入合同文本或上传文件，输出结构化风险报告、修改建议、替换条款、签署前清单。
- 合同生成：根据合同类型、交易背景、双方角色、保护倾向生成合同草案，并给出重点条款说明和待补充信息。
- 继续追问：报告生成后，用户可以围绕结果继续问“这个条款怎么改”“这里风险大不大”等问题。

当前已有 Dify 工作流“合同检查生成器”，主链路为：

用户输入 -> 意图识别与分流 -> 合同审查 / 合同生成 / 条款修订 / 意图澄清 -> 对应输出

## 2. 已确认产品边界

### 第一版必须做

1. 移动端优先，先跑通 H5 与 iOS 模拟器体验。
2. UI 不固定为“底部两个按钮”，先通过 3 套视觉原型选择最终产品形态。
3. 审查和生成并重。
4. 支持粘贴文本 + 上传 PDF / DOCX / TXT。
5. 图片拍照、OCR、手写识别放到第二版。
6. 结果采用“结构化报告 + 继续追问”。
7. 本地历史默认不保存，用户手动保存后才进入本地记录。
8. 输出必须包含“AI 辅助分析，不构成法律意见”。
9. 国内常见合同都做基础初筛，重点优化房屋租赁、劳动/用工、劳务合作。
10. 少量法条优先来自知识库引用；模型自带依据必须标注“未核验，需人工确认”。

### 第一版不做

1. 不做登录、会员、付费、云端同步。
2. 不做在线签署。
3. 不做律师撮合平台。
4. 不做团队审批流。
5. 不做应用内反馈系统；内测反馈线下沟通。
6. 当前没有 Apple Developer 账号，不把 TestFlight 多人真机分发作为第一轮硬验收。

## 3. 视觉与交互阶段

第一阶段先出 3 套手机端原型方向，每套包含：

- 首屏入口
- 合同输入页
- 审查/生成结果页

三套方向：

1. 专业极简：清爽、可信、效率工具感。
2. AI 智能助手：强调 AI 入口、对话感、智能分析感。
3. 法律文档工作台：更像移动端合同处理台，报告和文档感更强。

选定方向后，再决定最终入口结构，例如大输入框、卡片入口、对话式入口、浮动操作区或组合形式。入口不能在计划阶段硬编码为底部两个按钮。

## 4. 技术路线

推荐生产路线：

- 前端：uni-app Vue3 + TypeScript
- 首轮运行：H5 + iOS 模拟器
- 后续打包：iOS、Android、鸿蒙
- 后端：轻量云函数代理 Dify，优先 uniCloud
- AI 后端：Dify Workflow API
- 本地存储：手动保存到本地历史

后端必须隐藏 Dify API Key。APP 前端只调用云函数，不直接调用 Dify。

后续如果要正式 iOS 多人内测，需要 Apple Developer 账号并走 TestFlight；当前阶段先不阻塞在这一步。

## 5. APP 功能结构

### 输入能力

- 粘贴合同文本
- 上传 PDF / DOCX / TXT
- 选择合同类型
- 选择用户立场：甲方 / 乙方 / 中立 / 未知
- 选择审查重点：付款、交付、违约、知识产权、保密、解除、争议解决等
- 说明生成需求：合同类型、双方角色、交易背景、保护倾向、特殊约定

### 输出能力

- 总体结论
- 风险等级：低 / 中 / 高 / 未知
- 条款级风险列表
- 替换条款
- 合同草案
- 重点条款说明
- 签署前清单
- 少量依据与引用状态
- 继续追问入口
- 复制 / 导出
- 手动保存

## 6. Dify 工作流优化方向

Dify 保留一个总工作流，APP 传入 `mode` 和上下文，Dify 内部继续分流。

建议工作流结构：

1. 意图识别：只输出有限枚举 `review / generate / revise / clarify`。
2. 信息完整性检查：不足时先追问，不直接生成泛泛结果。
3. 审查分支：提取合同元信息、条款扫描、风险分级、替换条款。
4. 生成分支：生成合同正文、重点条款说明、待补充信息、签署清单。
5. 条款修订分支：原条款、问题、新条款、修改理由、偏向说明。
6. 依据处理：优先知识库引用；模型自带依据标注未核验。
7. JSON 输出整理：统一结构，方便 APP 渲染。

## 7. 开发里程碑

### 阶段 1：视觉方向选择

- 生成 3 套手机端视觉原型方向
- 用户选择一个方向，或要求融合多个方向
- 确认首屏、输入页、结果页的信息架构

验收：选定一个明确的视觉目标，作为后续 UI 实现依据。

### 阶段 2：uni-app 原型

- 搭建 uni-app Vue3 + TypeScript 项目
- 实现选定视觉方向的首屏、输入页、结果页
- 使用 mock 数据跑通审查、生成、追问、复制、导出、手动保存

验收：不依赖真实 Dify，也能完整演示合同处理流程。

### 阶段 3：Dify 接入

- 云函数封装 Dify API
- 接入文本输入
- 接入 PDF / DOCX / TXT 上传
- 支持真实 workflow 返回结构化结果
- 保存 `workflow_run_id`
- 处理失败、超时、空结果、格式异常

验收：APP 内提交文本或文件后，能看到真实 Dify 返回的结构化报告。

### 阶段 4：Dify 工作流专业化

- 统一四类分支输出 JSON
- 增强房屋租赁、劳动/用工、劳务合作样例
- 建小型法律知识库
- 规范法条引用与未核验标注
- 建测试集并反复跑样例

验收：常见合同样例输出稳定，风险分级、替换条款、签署清单可用。

### 阶段 5：内测准备

- H5 真机访问测试
- iOS 模拟器测试
- 错误提示和空状态打磨
- 内部使用说明
- 后续 TestFlight / Android / 鸿蒙打包准备

验收：可给内部人员演示和小范围试用。

## 8. 测试样例

### 合同生成

输入：生成一份软件开发服务合同，偏向保护甲方。  
预期：进入合同生成分支，输出合同正文、重点条款说明、签署前补充清单。

### 合同审查

输入：上传一份软件开发服务合同，用户为甲方，重点关注付款、交付、知识产权和违约责任。  
预期：输出总体风险、条款级风险、修改建议、替换条款和签署前清单。

### 房屋租赁

输入：审查一份房屋租赁合同，用户为承租人。  
预期：重点识别押金退还、维修责任、提前解约、违约金、转租、房屋交付状态。

### 劳动/用工

输入：审查一份劳动合同或用工协议。  
预期：重点识别试用期、工资、社保、加班、竞业限制、解除条件、违约责任。

### 劳务合作

输入：生成一份兼职劳务合作协议。  
预期：明确服务内容、交付成果、报酬结算、知识产权、保密、解除和违约责任。

### 意图澄清

输入：帮我生成一个合作合同。  
预期：不直接生成，先追问合同类型、双方角色、合作内容、付款方式、保护倾向。
